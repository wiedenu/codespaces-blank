Marketing Request Form · Aligning before we redesign

What are we actually solving for?

The form conversation keeps scattering because we're each solving a different problem. Before we change anything, one decision: which problem do we take on first. This frames that.

Start here — the north star

The form is ~5% of the work. 392 form requests vs. 7,773 total Jira tasks in the same window. Fewer than 1 in 20. Let's not over-index on this one channel.
Success = requests trending down. Fewer requests over time means stronger alignment across the org. A better-looking form doesn't automatically deliver that.

So the goal isn't a better form. It's better alignment. Every option below gets judged against that one question: does it move requests down?

The four ways we're seeing it

These aren't four opinions on one problem. They're four different problems — which is exactly why the room scatters. All four are legitimate; they just can't all be first.

Lens 1 · Routing

Get every request to the right place

"One smart form that routes anyone's request to the right person at the right time."

The problem being solved: requests go to the wrong place or stall. An orchestration problem.

Lens 2 · Friction

Fix the one step that's painful

"One small change would make this noticeably easier for the people using it every day."

The problem being solved: a specific daily friction point. A tactical-fix problem, raised by the people closest to the form.

Lens 3 · Simplification

Strip the form down

"The form asks too much. Cut it back to what's essential."

The problem being solved: the form is a burden that discourages good use. A complexity problem.

Lens 4 · Capacity

The real issue is intake & triage

"The form isn't the bottleneck — how we receive and route the work is. Ideally requests come in and someone triages."

The problem being solved: throughput. A capacity problem — with a real constraint: no staffing for a triage function today.

The one decision for the room

Which problem do we solve first? We don't have to pick one forever — but we sequence them instead of chasing all four at once. Choose the primary problem for this round; the rest become explicitly next-in-line or out of scope.

Tie-breaker when we disagree: which one most moves requests down / alignment up? That's the north star, and it keeps the choice about outcomes, not preferences.

How we'll run it

Before

I pre-wire each perspective 1:1 so we walk in aligned on the frame, not debating it live.

The session

30–45 min, one decision: the primary problem + the sequence for the rest. Not a form-design session.

After

Designing the actual form follows — once we know which problem it's serving.
